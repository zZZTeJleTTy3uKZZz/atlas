## 2. Проблема

Между треками нет единой PM-инфраструктуры. Конкретно:

- **Нет общей карты портфеля**. Ни в Notion, ни локально нет файла/дашборда, где видно «что у меня в работе / что остановилось / что умерло» по всем 4 трекам разом.
- **Memory Claude не знает про портфель.** В сессионной памяти есть только `project_cifro_new_products.md` (один проект). 10 клиентов, 14 утилит, 1 личный проект — не отражены вообще.
- **Разная зрелость документации.** Из 14 проектов в `Tests/`: 5 имеют полный `AGENTS.md + _project/docs`, 5 — только `README.md`, 2 — спецдоки (`PRD.md`, `REPORT.md`), 2 — бэкапы. AI-агент в каждый проект заходит с разной скоростью.
- **Notion и локальный markdown не сшиты.** В Notion есть проекты с B24-линками. В локальных `AGENTS.md` — нет ссылок обратно. Переключение треков = ручной контекст-свитчинг.
- **Нет ритуала обзора.** Weekly review, прогресс-трекинг, решения «продолжать / остановить / убить» — не формализованы.
- **Нет правила миграции «утилита → продукт»**. Когда что-то из `Tests/*` пора продуктизировать? Кто решает? По каким критериям?

## 3. Цель

Построить **личную PM-инфраструктуру** (методология + документация + инструменты + ритуалы), которая:

1. **Даёт одну карту портфеля** на все 4 трека — с состоянием, приоритетом, блокерами.
2. **Сшивает** Notion (клиенты + задачи) с локальным markdown (dev-портфель) с Bitrix24 (CRM) с NotebookLM (research).
3. **Делает AI-агентов first-class исполнителями**: любой проект портфеля можно за 2 минуты отдать агенту, не объясняя всё с нуля.
4. **Поддерживает переход утилита ↔ продукт** без ломки документации.
5. **Масштабируется до 2-3 человек**, если Cifro.pro наймёт ещё людей или Дмитрий решит продуктизировать систему.

## 4. Границы (что входит и что НЕ входит в scope NP-005)

**Входит**:
- Методология (как думать про портфель, lifecycle, приоритизацию, review).
- Стандарты документации (какие файлы должны быть в проекте и зачем).
- Карта источников истины (что живёт в Notion, что в markdown, что в B24, что в NotebookLM, что в memory).
- Набор skills/утилит для работы с портфелем (регистрация, статус, onboard, review).
- Интеграционный слой между Notion-проектами и локальными папками (двунаправленная сшивка).

**НЕ входит** (на этапе v1):
- Продуктизация наружу (публичный релиз, биллинг, онбординг чужих пользователей).
- Перенос существующих систем (Linear, Jira, ClickUp) — пробуем свою.
- Ручные процессы, которые не масштабируются через AI-агента.

## 5. Целевая архитектура в первом приближении

> Это стартовая гипотеза — точный дизайн соберётся после ответа deep research в `METHODOLOGY.md` и `ARCHITECTURE.md`.

**Слой 1 — Портфель.** Единый markdown-реестр `PORTFOLIO.md` (или аналог) с таксономией: `client-project`, `business-product`, `personal-utility`, `personal-project`. По строке на единицу: id, name, path, status, priority, owner, last-touched, links.

**Слой 2 — Проектный стандарт.** В каждом проекте: `AGENTS.md` (контекст для агента) + `_project/docs/` (SSOT) + lifecycle-тег + обратная ссылка на запись в `PORTFOLIO.md`.

**Слой 3 — Internal routing.** Notion SSOT для задач/клиентов/файлов (там где есть B24-линк); локальный markdown SSOT для dev-артефактов (код, архитектура, ресёрч). Мост — ID в AGENTS.md и ссылка в Notion-project-entry.

**Слой 4 — Ритуалы.** Weekly review (что сдвинулось, что заглохло, что убить), monthly review (перепроверка приоритетов), quarterly review (стратегия + миграция утилит в продукты).

**Слой 5 — AI-агент как исполнитель.** Claude Code / Codex работают в любом проекте портфеля по стандарту AGENTS.md. Skills типа `portfolio-register`, `portfolio-status`, `portfolio-onboard` доводят скорость до минуты.

**Слой 6 — Memory.** Моя (Claude) memory хранит индекс портфеля с 1-строчным описанием каждой единицы, ссылками на канонические доки и живыми ID NotebookLM-блокнотов.
