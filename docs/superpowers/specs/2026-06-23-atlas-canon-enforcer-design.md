## Цель

Сейчас «канон» проекта = три шаблона (`CANONICAL_README_TEMPLATE` / `CANONICAL_AGENTS_TEMPLATE` /
`CANONICAL_GITIGNORE_TEMPLATE` в `projects.py`), которые `_create_canonical_files` пишет на `project add`
**только если файла ещё нет**. Это «создать при рождении», но НЕ «поддерживать в каноне со временем»:

- Нет команды «проверь, что проект соответствует канону» (drift не виден).
- Нет понятия обязательных элементов кроме трёх файлов (нет `_project/docs/`, нет AGENTS-frontmatter,
  нет промт-блока Atlas из #211, нет «тесты если есть код»).
- Список канона захардкожен в Python-шаблонах; кастомизировать/расширять под классы проектов нельзя.
- Субагент `project-initializer` валидирует «на глаз», правил не накапливает — уроки теряются.

Задача — ввести **явный, конфиг-управляемый канон** + **энфорсер** (`check` / `--fix`, идемпотентный,
clikit-`--json`) + механизм **дообучаемости** (как канон эволюционирует со временем). Локально в Atlas
(канон на хаб не синкается — это свойство файловой системы проекта, не доменной сущности).

## Контекст (факты разведки, по коду 2026-06-23)

- **Текущий канон-генератор** — `projects.py::_create_canonical_files` (строки ~358-395): создаёт
  `README.md` / `AGENTS.md` / `.gitignore` из строковых шаблонов с `.format(**common)`. Идемпотентность
  наивная: `if path.exists(): continue`. Контента не сверяет, дрейф не чинит.
- **Флаг** `project add --canonical/--no-canonical` (default True) — точка вызова генератора.
- **AGENTS-шаблон** сейчас БЕЗ YAML-frontmatter (просто `# AGENTS.md — {name}` + проза). Промт-блока
  «используй Atlas CLI» в нём НЕТ (это #211).
- **`_project/docs/`** канона нет в генераторе. У самого Atlas структура развитая:
  `_project/docs/{PROJECT_LOG/, ARCHITECTURE/decisions/, SCALING_PRODUCT/{plans,products,specs}}`.
  Эталон «как должно быть» — навык `saas-project-operating-model` (AGENTS.md + `_project/docs` +
  `PROJECT_LOG` канонический слой + planner/backlog + task registry).
- **Субагент** `atlas:project-initializer` (`skills/atlas/agents/project-initializer.md`) — изучает
  папку, дополняет метаданные в БД, переписывает README в каноническом формате. Правил не копит.
- **Эпик** `atlas-canon-enforcer` goal: «Расширить project-initializer (глубже валидация, не захламлять
  корень); копилка правил/уроков code-advisor для структуры проектов».
- **Типы проектов** теперь конфиг-управляемы (spec `2026-06-23-atlas-type-canon-container`): у типа есть
  `storage_group`. Канон логично делать **функцией типа** (у `kit`/`service` канон строже, чем у `inbox`).
- **clikit**: `emit_data` / `--json` по умолчанию — отчёт энфорсера ляжет в этот контракт.

## Не входит в scope

- Переезд существующих 44 проектов «в канон» массово (это **шаг разноса** после реализации, отдельной
  пачкой, с подтверждением по каждому).
- Линт **содержимого** кода (qualiy gate, ruff/mypy) — энфорсер про **структуру и обязательные
  артефакты**, не про стиль кода.
- Синхронизация канон-статуса в ядро/Notion (локальная проверка ФС).
- Любая авто-правка истории git / авто-commit (энфорсер пишет файлы в рабочее дерево, коммитит человек).

## §1. Что такое «канон проекта»

**Канон** = набор **правил** (rules), каждое из которых проверяемо и (по возможности) авто-чинимо.
Правило = `{id, applies_to, severity, check, fix?}`.

### 1.1. Кандидатный список обязательных элементов (🟡 на согласование состава)

| id правила | Что требует | severity | applies_to (тип/условие) | auto-fix? |
|---|---|---|---|---|
| `agents-present` | `AGENTS.md` существует | error | все | да (из шаблона) |
| `agents-frontmatter` | в `AGENTS.md` есть YAML-frontmatter (`slug`, `type`, `status`, `atlas: true`) | error | все | да (вставка/мердж) |
| `agents-atlas-prompt` | промт-блок «используй Atlas CLI + вызывай навык atlas» (#211) | error | все | да (идемпотентная вставка) |
| `readme-present` | `README.md` существует | error | все | да |
| `gitignore-present` | `.gitignore` существует и содержит обязательный baseline (`.env`, `__pycache__`, …) | warn | все | да (мердж недостающих строк) |
| `project-docs-skeleton` | каталог `_project/docs/PROJECT_LOG/` существует | warn | все кроме `inbox` | да (создать каркас) |
| `project-log` | `_project/docs/PROJECT_LOG/PROJECT_STATE.md` существует | warn | все кроме `inbox`/`test` | да (заглушка) |
| `tests-if-code` | есть `tests/` ИЛИ `test_*.py`, ЕСЛИ в проекте есть код (`pyproject.toml`/`package.json`) | warn | `kit`/`service`/`superskill`/product | нет (нельзя нагенерить осмысленные тесты) |
| `no-root-clutter` | в корне нет «мусора» (дампы, `*.zip`, `output/`, временные) вне `.gitignore` | info | все | предложение, не авто-rm |
| `clikit-canon` *(опц.)* | если CLI — структура clikit (build_root_app) | info | `kit`/`service` | нет |

**Терминология severity:** `error` — нарушение канона (проект «не в каноне»); `warn` — желательно;
`info` — наблюдение/подсказка. Exit-код энфорсера в check-режиме: ≠0, если есть хоть один `error`
(порог настраивается, см. §2).

### 1.2. Откуда берётся список (🟡 ключевое решение)

Предложение — **встроенный baseline + слой пользовательских правил** (как сделали с типами проектов):

- **Base rules** — упакованный дефолт (Python-данные или `data/canon_rules.toml`): минимальный канон,
  который держит сам Atlas. Сюда переезжают текущие 3 шаблона + правила #211/`_project/docs`.
- **User rules** — `~/.atlas/canon.toml` (override через env `ATLAS_CANON_FILE`). Опционально.
  **Merge by rule-id** (user переопределяет severity/добавляет правила). Туда же — `[per_type]`-секции
  (канон строже для `kit`/`service`, мягче для `inbox`/`test`).
- **Шаблоны файлов** (README/AGENTS/gitignore/PROJECT_STATE) — выносятся из Python-литералов в
  `data/canon_templates/` (или в тот же toml), чтобы их можно было править без релиза кода.

Альтернатива (🟡): канон цел��ком в коде (проще, но негибко) — отвергаем, т.к. эпик прямо требует
«копилку правил» и кастом под классы проектов.

## §2. Поведение энфорсера

### 2.1. Команды (предложение имён, 🟡)

```
atlas project canon-check <ref> [--fix] [--strict] [--rule <id>...]   # один проект
atlas canon audit [--fix] [--type <slug>] [--strict]                  # весь портфель
atlas canon rules [list|show <id>|explain <id>]                       # что вообще проверяется
```

- `canon-check <ref>` — проверить ОДИН проект (по slug/UUID, как везде в Atlas).
- `canon audit` — пройтись по всем активным (не archived) проектам портфеля; агрегированный отчёт.
- `canon rules` — интроспекция действующего канона (base+user merged) — чтобы видеть, что энфорсит.

### 2.2. Режимы

- **check-only** (default): только диагностика, ФС не трогает. Exit ≠0 если есть `error`
  (с `--strict` — также при `warn`).
- **auto-fix** (`--fix`): применяет `fix` тех правил, у которых он есть и безопасен. Пишет в рабочее
  дерево проекта (файлы/каталоги). **Никогда** не удаляет (no-root-clutter только репортит), **никогда**
  не коммитит — это делает человек/daily-backup. После `--fix` повторный проход обязан давать «зелено»
  по всем auto-fixable правилам.
- `--rule <id>` — выборочный прогон одного/нескольких правил.

### 2.3. Контракт вывода (clikit `--json` по умолчанию)

```jsonc
{
  "ref": "playwright-network-dump",
  "canon_version": "1",
  "passed": false,
  "summary": {"error": 1, "warn": 2, "info": 1, "fixed": 0},
  "rules": [
    {"id": "agents-atlas-prompt", "severity": "error", "status": "fail",
     "message": "В AGENTS.md нет промт-блока Atlas", "fixable": true, "fixed": false},
    {"id": "readme-present", "severity": "error", "status": "pass"}
  ]
}
```
`audit` — то же, обёрнутое в массив `projects[]` + портфельный `summary`. `--text` — человекочитаемая
таблица (clikit `emit_table`). Каждая операция fix логируется в `action_log` (`canon_fixed`, details:
project + список применённых правил) — для аудита, что энфорсер реально менял.

### 2.4. Идемпотентность (инвариант)

- Каждое `fix` — **идемпотентно по содержимому**, а не по «файл существует». Промт-блок Atlas (#211)
  вставляется по маркеру (`<!-- atlas:canon:prompt -->…<!-- /atlas:canon:prompt -->`); повторный fix
  не дублирует. Gitignore — мерж недостающих строк (set-семантика), не дописывание дублей. Frontmatter —
  мерж ключей, не перезатирание чужих.
- Это исправляет текущую слабость `_create_canonical_files` (наивный `path.exists()`): энфорсер должен
  чинить **частично-канонные** проекты, а не только пустые.

### 2.5. Связь с текущим `project add`

`project add --canonical` остаётся, но внутри **переиспользует** канон-движок (вызывает те же `fix`
вместо отдельной ветки шаблонов). Так «создание» и «поддержание» канона — один источник правды;
шаблоны не расходятся.

## §4. Дообучаемость (🟡 ГЛАВНЫЙ открытый вопрос — нужно решение Дмитрия)

«Дообучаемость» = как канон **эволюционирует**: когда мы замечаем «о, у хорошего проекта так-то, а
правило этого не ловит» — как это знание попадает в энфорсер, не превращаясь в ручной хардкод. Три
подхода, по нарастанию амбиций:

### Вариант A — Конфиг-правила + копилка уроков (декларативно, без «обучения»)

Канон = `canon.toml` (§1.2). «Дообучение» = человек/агент **дописывает правило или урок** в конфиг.
Дополнительно — **журнал уроков** `~/.atlas/canon_lessons.md` (или таблица `canon_lessons`): когда
`project-initializer`/code-review ловит структурную проблему, он добавляет запись `{дата, проект,
наблюдение, предлагаемое правило}`. Периодически (ручной триггер) уроки превращаются в правила.

- **+** Просто, прозрачно, версионируемо в git, без LLM-затрат и без «магии».
- **+** Прямо ложится на формулировку эпика («копилка правил/уроков code-advisor»).
- **−** «Обучение» ручное: кто-то должен промотировать урок → правило.

### Вариант B — Вывод канона из «эталонных» проектов (learn-by-example)

Помечаем 2-3 проекта как **reference** (`atlas project canon-mark-reference <ref>`). Команда
`atlas canon learn` сканирует эталоны, выводит общие структурные инварианты (какие файлы/каталоги/секции
есть у ВСЕХ эталонов) и предлагает их как кандидаты-правила (severity=info по умолчанию, человек
промотирует до warn/error).

- **+** Канон отражает реальную «лучшую» практику портфеля, а не теорию.
- **−** «Общее у эталонов» легко даёт мусорные правила (случайные совпадения); нужна ручная фильтрация —
  то есть всё равно human-in-the-loop.
- **−** Сложнее реализовать; риск переусложнения для портфеля из ~44 проектов.

### Вариант C — Накопление исключений + LLM-адвайзор (адаптивно)

Энфорсер копит **исключения** (`atlas canon ignore <project> <rule> --reason`) — проект легально
отступает от правила. Плюс опциональный LLM-проход (`project-initializer`/code-advisor) предлагает
НОВЫЕ правила на основе прочитанного кода и накопленных исключений («правило X нарушается у 8 проектов
с тем же reason → возможно, правило надо смягчить/специализировать по типу»).

- **+** Канон самонастраивается под реальность; меньше «ложных error».
- **−** LLM-затраты, недетерминизм, риск «канон уплывает».
- **−** Самый дорогой; легко превратить в неподдерживаемую эвристику.

### Рекомендация (🟡 на утверждение)

**A сейчас, с заделом под C.** Делаем `canon.toml` + `canon_lessons` + механизм исключений
(`canon ignore` с reason — это дёшево и сразу полезно). Вывод правил из эталонов (B) и LLM-предложения
(C) — **отдельная волна**, только если ручная копилка (A) на практике окажется узким местом. Причина:
портфель один пользователь, ~44 проекта — ручная промоция уроков реалистична и прозрачна; «обучение»
без human-gate на таком масштабе даёт больше шума, чем пользы.

**Что должен решить Дмитрий:** (1) хватает ли A на старте; (2) нужен ли механизм `canon ignore`
(легальные исключения) в первой версии; (3) где хранить уроки — markdown-журнал vs таблица БД.

## §5. Связь с соседними механизмами

- **#211 (промт-блок Atlas)** — становится **правилом канона** `agents-atlas-prompt` (а не отдельной
  логикой в `project add`). Его fix = идемпотентная вставка по маркеру в `AGENTS.md` (+ `CLAUDE.md` при
  наличии). Таким образом #211 реализуется как первое «настоящее» правило энфорсера и валидирует движок.
- **Онбординг (`project add --canonical`)** — переиспользует канон-движок (§2.5). Новый проект сразу
  «в каноне» через те же `fix`, что чинят старые.
- **Субагент `project-initializer`** — после правки метаданных вызывает `canon-check <slug> --fix`
  (или энфорсер вызывает его для «умных» правил, требующих чтения проекта). Разделение: энфорсер —
  **детерминированные структурные** правила; субагент — **семантика** (правильный ли one-line, тип,
  переписать README по сути). Уроки субагента → копилка (§4-A).
- **Типы проектов** (spec type-canon-container) — `applies_to` правил ссылается на `type.slug` /
  `storage_group`; канон строже для `kit`/`service`/`superskill`.

## §6. План реализации (по кускам, каждый — зелёные тесты)

1. **Движок правил (pure-logic, без typer).** Модуль `src/atlas/pm/canon.py`: тип `Rule`, реестр,
   загрузка base + `~/.atlas/canon.toml` (merge by id), функции `check_project(path, project) ->
   list[RuleResult]` и `fix_project(...)`. Тесты: на временной папке-фикстуре правила корректно
   pass/fail; merge user-конфига; идемпотентность fix (двойной прогон → no-op).
2. **Правило `agents-atlas-prompt` (#211).** Маркер-вставка в AGENTS.md/CLAUDE.md; идемпотентно.
   Закрывает #211 поверх движка. Тесты: вставка, повторный прогон не дублирует, мерж с существующим
   frontmatter.
3. **Базовый набор правил** (present-файлы, gitignore-baseline, `_project/docs`-каркас, frontmatter).
   Шаблоны выносятся из `projects.py` в `data/canon_templates/`. Тест: старый `_create_canonical_files`
   заменён на движок, регрессия `project add --canonical` зелёная.
4. **CLI** `project canon-check` / `canon audit` / `canon rules` (clikit, `--json`, `--fix`, `--strict`).
   Тесты: exit-коды, json-контракт, `--fix` чинит и логирует в action_log.
5. **Дообучаемость A** — `canon_lessons` (журнал) + `canon ignore <project> <rule> --reason`. Тесты:
   исключение исключает правило из fail; урок добавляется/читается.
6. *(Вне scope первой волны)* B (learn-by-example) и C (LLM-адвайзор) — отдельной волной по решению §4.

### Вне scope (повтор для ясности)
Массовый ретрофит 44 проектов, линт кода, синк канон-статуса в ядро, авто-commit/авто-rm.

## §7. Обработка ошибок и краевые случаи

- `canon.toml` с битым rule-id/severity → понятная ошибка при загрузке, не падать молча.
- `--fix` на проекте без `local_path` в БД (или папка отсутствует) → skip + warn в отчёте, не падать.
- Правило `tests-if-code` не имеет fix — в `--fix` помечается `fixable:false`, остаётся в отчёте.
- Параллельный запуск `canon audit --fix` и ручной правки файлов — энфорсер не блокирует ФС; идемпотентность
  fix гарантирует безопасность повторов.
- `no-root-clutter` — НИКОГДА не удаляет; только репортит кандидатов (severity=info).
